# Workability Fixes Applied

Date: 2026-03-15

This document records the concrete fixes applied for the previously identified run-time/workability issues.

## 1) Broken STT endpoint (`/api/v1/speech-to-text`)

Problem:
- Endpoint called transcription with `None` instead of an audio file path, causing `TypeError` at runtime.

Changes:
- Updated endpoint to accept uploaded audio (`UploadFile = File(...)`).
- Wrote upload to `temp_data/` with a unique filename.
- Passed actual temp file path to STT service.
- Added cleanup in `finally`.
- Added `HTTPException(500)` for failures.

Files changed:
- `Backend/api/stt.py`

Outcome:
- Endpoint no longer crashes from `None` input path.
- Missing file request now correctly returns 422 validation error from FastAPI.

## 2) Broken RAG endpoint (`/api/v1/rag`)

Problem:
- Route passed only `question` to `rag_query`, but service requires `(question, user_answer)`.
- This caused runtime errors.

Changes:
- Extended request schema with `user_answer: str = ""` default.
- Updated route to call `rag_query(request.question, request.user_answer)`.

Files changed:
- `Backend/schemas/rag_schemas.py`
- `Backend/api/rag.py`

Outcome:
- Endpoint call contract now matches service signature.
- Route returns JSON response with `answer` key instead of failing.

## 3) Assessment voice endpoint error signaling (`/api/assessment/assess-voice`)

Problem:
- On internal failure, endpoint returned a JSON dict with HTTP 200 instead of error status.
- Frontend treated any 2xx as audio blob, causing incorrect behavior.

Changes:
- Converted exception path to `raise HTTPException(status_code=500, detail=...)`.

Files changed:
- `Backend/api/assessment.py`

Outcome:
- Backend failures now produce non-2xx status.
- Frontend `response.ok` check can reliably detect error responses.

## 4) Audio retrieval path mismatch (`/api/assessment/audio/{filename}`)

Problem:
- Audio retrieval route read from `generated_audio/`.
- TTS generation writes files into `audio/`.
- Result: route could not find files generated by current TTS flow.

Changes:
- Switched `AUDIO_DIR` from `generated_audio` to `audio`.
- Replaced JSON 200 error response with proper `HTTPException(404)`.

Files changed:
- `Backend/api/audio.py`

Outcome:
- Route now points to the same directory used by TTS generation.
- Missing files return correct 404.

## 5) Pinecone dependency mismatch across services

Problem:
- AI service pinned `pinecone-client==6.0.0` while backend used `pinecone==7.3.0`.
- Code imports use modern API (`from pinecone import Pinecone, ServerlessSpec`).
- This could break fresh installs depending on install order.

Changes:
- Replaced `pinecone-client==6.0.0` with `pinecone==7.3.0` in AI service requirements.

Files changed:
- `AI_Service/requirements.txt`

Outcome:
- Dependency alignment with code import style and backend requirement.
- Reduced environment-dependent startup/indexing failures.

## 6) Frontend route import casing mismatch

Problem:
- App imported `./pages/Dashboard` but file is `DashBoard.jsx`.
- Can break on case-sensitive filesystems.

Changes:
- Updated import path to match existing file name exactly.

Files changed:
- `Frontend/skillsetu-frontend/src/App.jsx`

Outcome:
- Cross-platform-safe module resolution for that route import.

## 7) Duplicate vision analyzer logic (`analyzer.py` vs `analyser.py`)

Problem:
- Two files had duplicate implementation, creating redundancy/confusion risk.

Changes:
- Kept `analyzer.py` as canonical implementation.
- Converted `analyser.py` into a compatibility shim that re-exports `evaluate_competency` from `analyzer.py`.
- Updated backend assessment import to canonical `analyzer` module.

Files changed:
- `AI_Service/src/vision/analyser.py`
- `Backend/api/assessment.py`

Outcome:
- Duplicate logic removed without breaking old import paths.
- Single source of truth for competency evaluation logic.
